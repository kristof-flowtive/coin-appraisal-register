Executive Summary

The M1 build is mid-flight on Glide, the no-code platform originally selected. During implementation we hit a pricing constraint that wasn't visible at design time: Glide meters every data change against a monthly "updates" quota, with overage charges that scale unfavourably with daily usage. At expected floor-staff volumes (50–100 customer interactions per day), Glide's effective cost lands in the $240–570/mo range — substantially higher than originally scoped.

This document evaluates three paths forward and recommends a decision. The Make.com workflows, Airtable schema, and metals-api integration are complete and unaffected by this decision — only the iPad frontend platform is on the table.

Background — what the app does

A floor-staff iPad app for live coin and bullion pricing. Reps select themselves from a dropdown, add line items (silver coins by count, bullion by gram weight), and see a live total computed from current spot prices and per-category margins.

The system has three layers, two of which are already complete:

LayerStatusWhat it does
AirtableCompleteStores config — coin types, margins, reps. Future deal log lives here too.
Make.comCompleteTwo scenarios: bulk-calc (live spot price + math) and config-load (read coin types + reps). Both tested and working.
iPad frontend (PWA)Decision pendingThe screen reps interact with — installs to iPad home screen, no app store. This document chooses the platform for this layer.

Why we're at this decision point

Glide bills "updates" — each data change (cart adds, calculations, refreshes) consumes from a monthly quota. After exhausting the included quota, every additional update costs $0.02.

What that means for floor-staff use

A typical customer interaction consumes ~9 updates (3 cart adds + 1 calculate + 3 cart clears + ~2 misc). Plus ~50 per daily config refresh. Projected monthly cost on each Glide tier:

The metering's shape is the issue. The cost rises with how much the app is actually used — exactly the wrong shape for a tool designed for daily, frequent floor use. Successful adoption makes the cost worse.

Option A — Glide (Business tier)

Stay on Glide; upgrade to Business tier ($249/mo base)

Continue the build already in progress, upgraded to Business tier for headroom and full feature access (Call API, native Airtable integration, Glide API).

Pros

  • Most build work already complete — fastest to production
  • Best mobile-app feel out of the box — Glide is mobile-first
  • Polished iPad PWA install (icon, splash, full-screen)
  • Real-time UI reactivity — taps update screen instantly
  • Native Airtable integration unlocked (eliminates one Make scenario)
  • Client can edit screens themselves later
  • Lowest-friction long-term maintenance

Cons

  • Highest recurring cost: $249/mo base + per-use overage
  • Cost rises with usage — busy days are expensive days
  • At 100 interactions/day: ~$570/mo
  • M2 (deal logging) will increase update consumption further
  • Locked into Glide's component library and constraints
  • Platform lock-in — migrating away later is a full rebuild
Year-1 estimate at 50 interactions/day: ~$4,400  |  At 100/day: ~$6,800

Option B — Softr

Migrate to Softr (Basic or Professional tier)

Switch to a competing no-code platform with native Airtable integration on lower tiers and fixed-tier pricing without Glide-style per-action metering for typical app use.

Pros

  • Substantially lower recurring cost ($59–167/mo)
  • No per-update meter punishing app usage
  • Native Airtable on entry tier (eliminates config-load Make scenario)
  • Still no-code — client can edit screens later
  • Existing Make scenarios work as-originally-designed

Cons

  • Real-time reactivity gap — Calculate Total round-trip likely 7–17 seconds (vs ~2s on Glide) due to Airtable polling delays. This is the largest concrete UX risk for a floor tool.
  • Less polished mobile-app feel — Softr is portal-oriented; PWAs feel like installed websites rather than native apps
  • Touch-optimized components are weaker — finger-friendly tap targets need custom CSS
  • Throw away Glide work to date (small loss)
  • Workflow action metering exists; needs validation against floor-staff volume
  • Mobile-app maturity lags Glide
Year-1 estimate: ~$700 (Basic) to ~$2,000 (Professional)

Option C — Custom-built PWA

Develop a purpose-built progressive web app

Build a custom iPad PWA using modern web technologies (React + Tailwind), deployed on Vercel free tier. Functions identically to Glide/Softr in user experience but is a custom-coded application with no platform constraints.

Pros

  • Zero recurring cost — Vercel free tier handles this scale
  • Full UX control — finger-perfect tap targets, custom keypads, native feel
  • Fastest possible Calculate Total round-trip (~2s)
  • No platform metering, ever
  • Cheapest option after build cost is amortized
  • Architecture is durable through M2 and beyond — no platform lock-in
  • Existing Make scenarios work as-originally-designed

Cons

  • One-time build cost — ~16–32 hours of development time
  • Client cannot edit screens themselves — every change requires developer involvement
  • Maintenance burden owned by you/dev (dependency updates, hosting)
  • Initial GitHub + Vercel setup required (one-time, ~30 min)
  • Loss of no-code editability is a permanent trade-off
Year-1 estimate: build cost (one-time, ~$800–$3,200 at typical dev rates) + ~$0/mo recurring

Recommendation

The decision narrows to two viable paths, distinguished by which trade-off matters more:

Choose Glide Business ($249/mo + overage) if:

  • The fastest path to production matters most
  • Self-edit ability for the client is high-value (they'll tweak screens themselves)
  • Recurring cost is acceptable as part of operating expense
  • Volume is expected to stay low to moderate (under 50 interactions/day)
  • Client values "no engineering dependency" highly

Choose Custom PWA if:

  • Long-term cost discipline matters more than self-edit ability
  • Client trusts the dev/agency relationship for future changes
  • Volume is expected to be moderate to heavy (50+ interactions/day) — overage savings recoup build cost quickly
  • App horizon is medium-to-long — custom PWA's economics improve over time
  • Client values polish and UX control highly (custom keypads, gesture support, etc.)

Why Softr is not recommended

The reactivity gap on Calculate Total is the single most concrete UX risk for a tool used live in front of a customer. The cost savings vs Glide ($150–400/mo) don't fully compensate for that downgrade, and the build effort to mitigate UX issues is comparable to the custom PWA route — without the polish payoff. Softr is a strong platform for portals and internal tools, but it's the wrong fit for a fast, daily-use floor calculator on iPad.

What we need from the client to decide

  1. Expected daily usage volume — number of customer interactions per day, on average and on busy days. Confirms cost projections under each option.
  2. Recurring vs one-time cost preference — is "$249–570/mo forever" easier to absorb than "one-time build cost now, near-zero after"?
  3. Self-edit priority — does the client want to modify screens themselves without involving us, or are they comfortable routing changes through us?
  4. Time horizon — how long is this app expected to be in use? (Custom PWA's economics improve significantly over multi-year horizons.)
  5. Client's tolerance for the M1 timeline — Glide finishes ~1 day faster than Custom PWA. Does that matter?

Once the client weighs these five points, the choice between Glide Business and Custom PWA should be clear-cut. Either is a defensible production choice; the right answer depends on which trade-off the client prefers.
